# Pending Features (via RedisVL)

> **Status: PENDING.** The features on this page are **not yet
> implemented** in pyredis-om. They will land with the upcoming
> [RedisVL integration](/redisvl), which delegates RediSearch/Vector
> index operations to the dedicated
> [`redisvl`](https://github.com/redis/redis-vl) library — the same
> path taken by [upstream PR #791](https://github.com/redis/redis-om-python/pull/791)
> and the upstream [RFC #790](https://github.com/redis/redis-om-python/issues/790).

This page documents *what's planned* so users can preview the API and
track progress. It will move to its own full reference once the
integration lands.

## Why delegate?

Building parallel `FT.*` support in pyredis-om would duplicate the
maintenance burden of redisvl (which already covers hybrid search,
SVS-VAMANA, vector policies, and more). The upstream RFC explicitly
asks for this delegation:

> We should delegate vector and FT.HYBRID support to the redisvl
> library, which is the canonical Python implementation of these
> features.
>
> — [redis/redis-om-python#790](https://github.com/redis/redis-om-python/issues/790)

Concretely, this means:

- **What pyredis-om keeps**: Pydantic model definition, schema
  generation, query building for the existing operators, low-level
  data-structure wrappers (Streams, Vector Sets, Bitmaps, ...).
- **What redisvl owns**: Index management, vector field configuration,
  hybrid search (`FT.HYBRID`), aggregations (`FT.AGGREGATE`), SVS-VAMANA
  graph index, and other advanced search features.

## Planned feature list

### `FT.HYBRID` — hybrid BM25 + vector queries

Combines full-text BM25 ranking with vector similarity in a single
query. Useful for "find me semantically similar documents that also
mention this keyword."

```python
# PLANNED — do not use yet.
from redisvl.query import HybridQuery

query = HybridQuery(
    text="redis hash field expiration",
    vector=embed("redis hash field expiration"),
    text_field="body",
    vector_field="embedding",
    alpha=0.5,  # 0 = pure vector, 1 = pure BM25
)
results = index.query(query)
```

Status: **pending redisvl delegation.** Tracked by upstream issue
[#258](https://github.com/redis/redis-om-python/issues/258).

### SVS-VAMANA vector index (`VSIM IN` for `FT.HYBRID`)

Redis 8.8 introduces the SVS-VAMANA graph-based vector index, an
alternative to HNSW with better compression and recall at scale.
redisvl exposes this as a vector index algorithm option.

```python
# PLANNED — do not use yet.
from aredis_om import JsonModel, VectorFieldOptions


class Article(JsonModel):
    body: str = Field(index=True, full_text_search=True)
    embedding: list[float] = Field(
        index=True,
        vector_field=VectorFieldOptions(
            algorithm="svs-vamana",
            dim=384,
            compression="LVQ8",  # or "LeanVec", "GlobalPQ"
        ),
    )
```

Status: **pending redisvl delegation.**

### `FT.AGGREGATE` — server-side aggregations

Performs grouping, reduction, sorting, and limiting server-side,
returning only the projected fields. Useful for dashboards and
faceted navigation.

```python
# PLANNED — do not use yet.
from redisvl.query import AggregateQuery

q = AggregateQuery("article").group_by(["@category"]).reduce(
    count=Count(), avg_score=Avg("@score")
).sort_by(AvgField("@score"), desc=True).limit(0, 20)
results = index.aggregate(q)
```

Status: **pending redisvl delegation.** Tracked by upstream issue
[#258](https://github.com/redis/redis-om-python/issues/258).

### Vector field policies

Custom pre-filtering, post-filtering, and runtime swap-in of vector
index parameters (algo, compression, search-time `EF`).

Status: **pending redisvl delegation.**

### `AsyncSearchIndex` lifecycle hooks

Wrappers that connect the existing `Meta.database` model pattern to
redisvl's `AsyncSearchIndex.from_existing(...)` so user code doesn't
have to rebuild the index from Pydantic models.

Status: **pending redisvl delegation.**

## How to track progress

- [PR #791](https://github.com/redis/redis-om-python/pull/791) — the
  reference integration on upstream.
- Issue [#790](https://github.com/redis/redis-om-python/issues/790) —
  the upstream RFC for delegation.
- Issue [#258](https://github.com/redis/redis-om-python/issues/258) —
  `FT.AGGREGATE` / hybrid search.

The integration is expected to land in the next minor release of
pyredis-om after the current non-search-feature batch ships.

## In the meantime

If you need these features today:

1. Install [`redisvl`](https://pypi.org/project/redis-vl/) directly
   and use it standalone (it doesn't depend on redis-om).
2. Use [`redisvl` indexes](https://docs.redisvl.com/) for hybrid
   search and advanced vector work, while keeping pyredis-om for
   Pydantic models, persistence, and the operators/queries API.

Once the integration lands, both will share the same `Meta.database`
connection.

## Valkey Module Support

[Valkey](https://valkey.io/) is the open-source Redis fork. pyredis-om runs
against it today for `HashModel` CRUD and a subset of query features, but
**Valkey is not a drop-in replacement for Redis Stack** — valkey-search has
several intentional design differences from RediSearch that affect query
behaviour.

**What works:**

- **`valkey/valkey:9-alpine`** (no modules): `HashModel` CRUD, validation,
  pipelines.
- **`valkey/valkey-bundle:alpine`** (with modules): `JsonModel` CRUD,
  `FT.CREATE` with `NUMERIC`/`TAG`/`TEXT`/`VECTOR`, explicit-filter queries,
  plain `FT.AGGREGATE`, KNN vector search.
- The existing `has_redis_json()` and `has_redisearch()` checks work against
  Valkey because the module commands (`JSON.SET`, `FT.SEARCH`) are identical.
- The RESP3 `FT.INFO` shim handles Valkey's flat-list response shape.

**What does NOT work (empirically verified against valkey-search 1.2.0):**

- `FT.SEARCH '*'` (match-all query) is rejected by design — see
  [valkey-search #283](https://github.com/valkey-io/valkey-search/pull/283).
  This breaks every `Model.find()` with no filter arguments.
- Dual `TAG` + `TEXT` on the same source field (used by
  `Field(index=True, full_text_search=True)`) is rejected as a duplicate.
- `GEO` field type (used by `Coordinates`) is not supported.
- `FT.ALIASUPDATE` and the entire zero-downtime migration path are not
  supported.
- `FT.AGGREGATE WITHCURSOR` (used by `FindQuery.iter_cursor()`) is not
  supported.
- Several aggregation reducers (`QUANTILE`, `TOLIST`, `matched_terms`,
  `FIRST_VALUE`, `RANDOM_SAMPLE`, `COUNT_DISTINCTISH`) are not supported.

Switching to `valkey-py` does **not** fix any of these — they are
server-side limitations.

See [Upcoming Valkey Support](/upcoming_valkey) for the full empirically
verified findings, reproduction commands, and the open design decisions for
handling each gap.
